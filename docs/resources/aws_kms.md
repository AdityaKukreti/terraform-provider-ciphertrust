# Resource "ciphertrust_aws_kms" (terraform-provider-ciphertrust)

Use this resource to create and manage KMS keys for AWS accounts in CipherTrust Manager. If the KMS is not found during refresh it is removed from state automatically.

## Example Usage

```terraform
# This resource is dependent on a ciphertrust_aws_connection resource
resource "ciphertrust_aws_connection" "aws_connection" {
  name = "aws_connection_name"
}

# Define a kms resource without using the ciphertrust_aws_account_details data-source and assign it to the connection
resource "ciphertrust_aws_kms" "kms" {
  account_id     = "aws-account-id"
  aws_connection = ciphertrust_aws_connection.aws_connection.id
  name           = "kms-name"
  regions        = ["aws-region", "aws-region"]
}

# Define a kms resource using the ciphertrust_aws_account_details data-source and assign it to the connection
data "ciphertrust_aws_account_details" "account_details" {
  aws_connection = ciphertrust_aws_connection.aws_connection.id
}

resource "ciphertrust_aws_kms" "kms_with_details" {
  account_id     = data.ciphertrust_aws_account_details.account_details.account_id
  aws_connection = ciphertrust_aws_connection.aws_connection.id
  name           = "kms-name"
  regions = [data.ciphertrust_aws_account_details.account_details.regions[0],
  data.ciphertrust_aws_account_details.account_details.regions[1]]
}

# Define an AWS key
resource "ciphertrust_aws_key" "aws_key" {
  kms_id = ciphertrust_aws_kms.kms.id
  region = ciphertrust_aws_kms.kms.regions[0]
}
```

## Import

An AWS KMS registration can be imported using its CipherTrust Manager resource
ID. The resource ID is the `id` attribute returned after creation and is also
visible in the CCKM interface.

### Important: Behaviour after import

After import, Terraform will attempt to make the KMS match your resource
configuration on the next `terraform apply`.

**`name` and `account_id` force replacement** - these two attributes use
`RequiresReplace`. If the values in your resource configuration do not exactly
match the imported KMS, Terraform will destroy the existing registration and
create a new one. Always inspect the imported state with
`terraform state show ciphertrust_aws_kms.example` and confirm both values are
correct before running `terraform apply`.

**`aws_connection`** - the API stores and returns the connection name. If your
resource configuration references the connection by its UUID (for example
`aws_connection = ciphertrust_aws_connection.my_conn.id`), Terraform will see
a permanent diff because the stored value is a name, not a UUID. Use the
connection name instead (for example
`aws_connection = ciphertrust_aws_connection.my_conn.name`) so that the config
value matches what the API returns.

**`regions`** - updatable. If your resource configuration specifies fewer
regions than the imported KMS has, Terraform will remove the missing ones on
the next apply. Use `ignore_changes` on `regions` until you are ready to
reconcile the list.

**`assume_role_arn` and `assume_role_external_id`** - optional and updatable.
These fields are only included in the update request when non-empty, so
omitting them from the resource configuration is safe if they are not in use.

To avoid unintended changes immediately after import, add a `lifecycle` block:

```terraform
lifecycle {
  ignore_changes = [
    assume_role_arn,
    assume_role_external_id,
    aws_connection,
    regions,
  ]
}
```

The recommended approach:

1. Add the `lifecycle { ignore_changes }` block shown above to the resource
2. Run `terraform apply` (or use `terraform import` for Terraform versions earlier than v1.5.0)
3. Run `terraform state show ciphertrust_aws_kms.example` to inspect the imported values
4. Update your resource configuration to match the imported state, paying particular attention to `name`, `account_id`, `aws_connection`, and `regions`
5. Remove `ignore_changes` so Terraform manages the attributes going forward

You can import a resource into Terraform in two ways.

### 1. Import using an import block and terraform apply

Only available for Terraform v1.5.0 and later.

When Terraform processes the `import` block it:

1. Reads the KMS from the API using the provided ID and stores the current values as state
2. Compares that state to your resource block configuration
3. Applies any differences as updates to the KMS (immediately during the same apply)

All three steps happen within a single `terraform apply` run.

**Step 1 - Import with `ignore_changes`** (safe starting point):

```terraform
import {
  to = ciphertrust_aws_kms.example
  id = "abc12345-1234-1234-1234-123456789012"
}

resource "ciphertrust_aws_kms" "example" {
  account_id     = "<aws-account-id>"
  aws_connection = ciphertrust_aws_connection.my_conn.name
  name           = "<kms-name>"
  regions        = ["<aws-region>"]
  lifecycle {
    ignore_changes = [
      assume_role_arn,
      assume_role_external_id,
      aws_connection,
      regions,
    ]
  }
}
```

**Step 2 - Final config after inspecting state** (remove `ignore_changes` once
values are correct):

```terraform
resource "ciphertrust_aws_kms" "example" {
  account_id     = "<aws-account-id>"
  aws_connection = ciphertrust_aws_connection.my_conn.name
  name           = "<kms-name>"
  regions        = ["<aws-region-1>", "<aws-region-2>"]
  # assume_role_arn        = "<role-arn>"        # include only if in use
  # assume_role_external_id = "<external-id>"    # include only if in use
}
```

### 2. Import using the terraform import command

When `terraform import` is run it:

1. Reads the KMS from the API using the provided ID and stores the current values as state
2. Writes the imported state to the state file - no plan or apply is performed

Before running `terraform apply`, use the `lifecycle { ignore_changes }` block
shown above to avoid unintended changes. Run `terraform plan` first to review
exactly what would be applied.

```shell
terraform import ciphertrust_aws_kms.example abc12345-1234-1234-1234-123456789012
```
