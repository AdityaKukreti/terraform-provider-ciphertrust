# Resource "ciphertrust_aws_xks_key" (terraform-provider-ciphertrust)

Use this resource to create and manage AWS XKS keys in CipherTrust Manager.

## Example Usage

```terraform
# Define an AWS connection
resource "ciphertrust_aws_connection" "aws-connection" {
  name = "connection-name"
}

# Get the AWS account details
data "ciphertrust_aws_account_details" "account_details" {
  aws_connection = ciphertrust_aws_connection.aws-connection.id
}

# Define a kms
resource "ciphertrust_aws_kms" "kms" {
  account_id     = data.ciphertrust_aws_account_details.account_details.account_id
  aws_connection = ciphertrust_aws_connection.aws-connection.id
  name           = "kms-name"
  regions        = data.ciphertrust_aws_account_details.account_details.regions
}

# Define an AES CipherTrust key for creating EXTERNAL_KEY_STORE with CipherTrust Manager as key source
# key should be unexportable, undeletable, symmetric AES 256 key.
resource "ciphertrust_cm_key" "cm_aes_key" {
  name         = "aes-key-name"
  algorithm    = "AES"
  usage_mask   = 60
  unexportable = true
  undeletable  = true
  # Setting remove_from_state_on_destroy to true will allow the key to be deleted from terraform state on destroy however, it will remain in CipherTrust Manager.
  remove_from_state_on_destroy = true
}

# Define unlinked external custom keystore with CipherTrust Manager as key source and PUBLIC_ENDPOINT proxy connectivity
resource "ciphertrust_aws_custom_keystore" "custom_keystore" {
  name                        = "keystore-name"
  region                      = "us-west-1"
  kms_id                      = ciphertrust_aws_kms.kms.id
  linked_state                = false
  connect_disconnect_keystore = "DISCONNECT_KEYSTORE"
  local_hosted_params = {
    blocked             = false
    health_check_key_id = ciphertrust_cm_key.cm_aes_key.id
    max_credentials     = 8
    source_key_tier     = "local"
  }
  aws_param = {
    xks_proxy_uri_endpoint = "https://demo-xksproxy.thalescpl.io"
    xks_proxy_connectivity = "PUBLIC_ENDPOINT"
    custom_key_store_type  = "EXTERNAL_KEY_STORE"
  }
}

# Define an unlinked XKS key with CipherTrust Manager as key source in above unlinked external key store
# Keys can only be linked once the keystore is linked
resource "ciphertrust_aws_xks_key" "xks_key" {
  local_hosted_params = {
    blocked             = false
    custom_key_store_id = ciphertrust_aws_custom_keystore.custom_keystore.id
    linked              = false
    source_key_id       = ciphertrust_cm_key.cm_aes_key.id
    source_key_tier     = "local"
  }
  aws_param = {
    alias = ["key-name"]
  }
}

# An example resource for importing an existing xks key
resource "ciphertrust_aws_xks_key" "imported_xks_key" {
  local_hosted_params = {
    blocked             = false
    custom_key_store_id = "0813e489-6930-4c4f-a9ab-85ff275f9122"
    linked              = false
    source_key_id       = "5b0cce40a9434708bfb2510a670dce2d12a0253bda444a109224e519f0df5619"
    source_key_tier     = "local"
  }
}
```

## Import

An AWS XKS key can be imported using its CipherTrust Manager key ID. The key ID
is returned in the API response body as the `id` field and is also visible in
the CCKM interface.

### Important: Behaviour after import

After import, Terraform will attempt to make the key match your resource configuration.

If updateable attributes are missing or incorrect they may be updated or removed during `terraform apply`.

The following attributes and blocks require particular attention after import:

**`local_hosted_params`** - this block is not returned by the API. After import
it is absent from Terraform state. Before running `terraform apply`, you must
configure the block to match the key's current state, or add `local_hosted_params`
to `ignore_changes`. If the block is present in your configuration but differs
from the key's current state, Terraform will attempt to apply those differences
on the next apply, which could trigger unintended block, unblock, or link
operations.

**`local_hosted_params.linked`** - setting this to `true` when the key is
currently unlinked triggers an irreversible link operation that registers the
key with AWS KMS. Once linked, it cannot be unlinked. Inspect the key's current
linked state with `terraform state show ciphertrust_aws_xks_key.example` and
configure this field carefully.

**`local_hosted_params.blocked`** - if the configured value differs from the
key's current blocked state, Terraform will block or unblock the key on the
next apply.

**Linked-state conditional behaviour** - updates to `description`, `key_policy`,
`alias`, `tags`, and `enable_key` are only applied in AWS when the key is in a
linked state (`linked = true` in `local_hosted_params`). For unlinked keys,
Terraform will accept the plan but those changes will not be made to the key.
Only `enable_rotation` and `local_hosted_params.blocked`/`.linked` take effect
for unlinked keys.

**`enable_rotation`** - not returned by the API; imported state shows an empty
block. If the key is scheduled for rotation, include the `enable_rotation` block
with the correct scheduler details so that Terraform takes ownership of the
rotation schedule. Omitting it is safe - the key remains scheduled for rotation
but Terraform will not manage it. Removing a previously configured
`enable_rotation` block during `terraform apply` unschedules the key for
rotation.

In particular, for list and map attributes such as `alias` and `tags`, omitting
them from the resource configuration may remove existing values when the key is
linked.

To avoid unintended changes to an existing key, do one of the following:

**Option 1 - Configure the resource block to match the key's current state**
before running `terraform apply`. Set `aws_param` (including `alias`,
`description`, and `tags`), `key_policy`, `enable_rotation`, `enable_key`, and
`local_hosted_params` to their current values so that Terraform detects no diff.

**Option 2 - Use `lifecycle { ignore_changes }` to suppress diffs** on
attributes you are not yet ready for Terraform to manage:

```terraform
lifecycle {
  ignore_changes = [
    aws_param,
    aws_param.description,
    aws_param.tags,
    enable_key,
    enable_rotation,
    key_policy,
    local_hosted_params,
  ]
}
```

The recommended approach (temporary use of `ignore_changes`):

1. Add the `lifecycle { ignore_changes }` block to the resource
2. Run `terraform apply` (or use `terraform import` for Terraform versions earlier than v1.5.0)
3. Run `terraform state show ciphertrust_aws_xks_key.example` to inspect the imported values
4. Update your resource configuration to match the imported state, paying particular attention to `local_hosted_params.linked` and `local_hosted_params.blocked`
5. Remove `ignore_changes` so Terraform manages the attributes going forward

You can import a resource into Terraform in two ways.

### 1. Import using an import block and terraform apply

Only available for Terraform v1.5.0 and later.

When Terraform processes the `import` block it:

1. Reads the key from the API using the provided ID and stores the current values as state
2. Compares that state to your resource block configuration
3. Applies any differences as updates to the key (immediately during the same apply)

All three steps happen within a single `terraform apply` run.

**Step 1 - Import with `ignore_changes`** (safe starting point):

```terraform
import {
  to = ciphertrust_aws_xks_key.example
  id = "abc12345-1234-1234-1234-123456789012"
}

resource "ciphertrust_aws_xks_key" "example" {
  local_hosted_params = {
    blocked             = false
    custom_key_store_id = "<custom-keystore-resource-id>"
    linked              = true
    source_key_id       = "<cm-source-key-id>"
    source_key_tier     = "local"
  }
  lifecycle {
    ignore_changes = [
      aws_param,
      aws_param.description,
      aws_param.tags,
      enable_key,
      enable_rotation,
      key_policy,
      local_hosted_params,
    ]
  }
}
```

**Step 2 - Final config after inspecting state** (remove `ignore_changes` once
values are correct):

```terraform
resource "ciphertrust_aws_xks_key" "example" {
  enable_key = true
  aws_param = {
    alias       = ["alias/my-xks-key"]
    description = "<description>"
    tags        = { Environment = "prod" }
  }
  local_hosted_params = {
    blocked             = false
    custom_key_store_id = "<custom-keystore-resource-id>"
    linked              = true
    source_key_id       = "<cm-source-key-id>"
    source_key_tier     = "local"
  }
  key_policy = {
    # set values from terraform state show output
  }
  enable_rotation = {
    disable_encrypt = false
    job_config_id   = "<scheduler-resource-id>"
    key_source      = "local"
  }
}
```

The example above shows all updateable attributes. Include only the attributes
that apply to your key - for example, omit `aws_param.tags` if the key has no
tags, or omit `enable_rotation` if the key is not scheduled for rotation.

### 2. Import using the terraform import command

When `terraform import` is run it:

1. Reads the key from the API using the provided ID and stores the current values as state
2. Writes the imported state to the state file - no plan or apply is performed

Before running `terraform apply`, apply Option 1 or Option 2 above to avoid
unintended changes. Run `terraform plan` first to review exactly what would be
applied.

```shell
terraform import ciphertrust_aws_xks_key.example abc12345-1234-1234-1234-123456789012
```
