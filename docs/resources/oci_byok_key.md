# Resource "ciphertrust_oci_byok_key" (terraform-provider-ciphertrust)

Use this resource to create and manage OCI BYOK keys in CipherTrust Manager.

## Example Usage

```terraform
resource "ciphertrust_cm_key" "test_key" {
  name       = "test-key-name"
  algorithm  = "RSA"
  usage_mask = 60
  key_size   = 2048
}

resource "ciphertrust_oci_byok_key" "test_key" {
  # Required parameters
  name          = "key-name"
  source_key_id = ciphertrust_cm_key.test_key.id
  vault         = "vault-id"
  oci_key_params = {
    compartment_id  = "compartment-ocid"
    protection_mode = "SOFTWARE"
    # Optional oci_key_params
    defined_tags = [
      {
        tag = "oci-namespace"
        values = {
          key-tag = "key-value"
        }
      }
    ]
    freeform_tags = {
      key-tag = "key-value"
    }
  }
  # Optional parameters
  source_key_tier = "local"
  enable_auto_rotation = {
    job_config_id = "ciphertrust.scheduler.scheduled.rotation_job.id"
    key_source    = "ciphertrust"
  }
  enable_key                 = true
  schedule_for_deletion_days = 14
}
```

## Import

An OCI BYOK key can be imported using its CipherTrust Manager key ID. The key
ID is returned in the API response body as the `id` field and is also visible
in the CCKM interface.

### Important: Behaviour after import

After import, Terraform will attempt to make the key match your resource
configuration.

If updateable attributes are missing or incorrect they may be updated or
removed during `terraform apply`.

**`source_key_id` and `source_key_tier`** - not returned directly by the
key API, but they are reconstructed after import by listing the key's versions
and reading the first version. After import these fields will be populated in
state. They are not updatable - changing them in configuration is silently
ignored by the update operation. To change the source key, the resource must
be destroyed and recreated.

**`vault` and `oci_key_params.protection_mode`** - also not updatable.
Changing either in configuration is silently ignored. To change these, the
resource must be destroyed and recreated.

**`enable_auto_rotation`** - not returned by the API; imported state shows
nil. If the key is scheduled for rotation, include the `enable_auto_rotation`
block with the correct scheduler details so that Terraform takes ownership.
Omitting it is safe - the key remains scheduled for rotation but Terraform
will not manage it. Removing a previously configured `enable_auto_rotation`
block during `terraform apply` unschedules the key for rotation.

To avoid unintended changes, do one of the following:

**Option 1 - Configure the resource block to match the key's current state**
before running `terraform apply`. Set `name`, `enable_key`,
`oci_key_params.compartment_id`, `oci_key_params.freeform_tags`, and
`oci_key_params.defined_tags` to their current values so that Terraform
detects no diff.

**Option 2 - Use `lifecycle { ignore_changes }` to suppress diffs** on
attributes you are not yet ready for Terraform to manage. The list below
suppresses diffs on the creation-time fields and the common updatable
attributes, while leaving `oci_key_params.compartment_id` manageable so
that Terraform can move the key to a different compartment when needed:

```terraform
lifecycle {
  ignore_changes = [
    enable_auto_rotation,
    enable_key,
    name,
    oci_key_params.defined_tags,
    oci_key_params.freeform_tags,
    oci_key_params.protection_mode,
  ]
}
```

The recommended approach (temporary use of `ignore_changes`):

1. Add the `lifecycle { ignore_changes }` block to the resource
2. Run `terraform apply` (or use `terraform import` for Terraform versions earlier than v1.5.0)
3. Run `terraform state show ciphertrust_oci_byok_key.example` to inspect the imported values
4. Update your resource configuration to match the imported state
5. Remove `ignore_changes` so Terraform manages the attributes going forward

You can import a resource into Terraform in two ways.

### 1. Import using an import block and terraform apply

Only available for Terraform v1.5.0 and later.

When Terraform processes the `import` block it:

1. Reads the key from the API using the provided ID and stores the current values as state
2. Compares that state to your resource block configuration
3. Applies any differences as updates to the key (immediately during the same apply)

All three steps happen within a single `terraform apply` run.

**Step 1 - Import with `ignore_changes`** (safe starting point):

```terraform
import {
  to = ciphertrust_oci_byok_key.example
  id = "abc12345-1234-1234-1234-123456789012"
}

resource "ciphertrust_oci_byok_key" "example" {
  name          = "<key-name>"
  vault         = "<vault-resource-id>"
  source_key_id = "<cm-source-key-id>"
  oci_key_params {
    compartment_id  = "<compartment-ocid>"
    protection_mode = "HSM"
  }
  lifecycle {
    ignore_changes = [
      enable_auto_rotation,
      enable_key,
      name,
      oci_key_params.defined_tags,
      oci_key_params.freeform_tags,
      oci_key_params.protection_mode,
    ]
  }
}
```

**Step 2 - Final config after inspecting state** (remove `ignore_changes`
once values are correct):

```terraform
resource "ciphertrust_oci_byok_key" "example" {
  name          = "<key-name>"
  vault         = "<vault-resource-id>"
  source_key_id = "<cm-source-key-id>"   # not updatable - set from imported state
  source_key_tier = "local"              # not updatable - set from imported state
  enable_key    = true
  oci_key_params {
    compartment_id  = "<compartment-ocid>"
    protection_mode = "HSM"              # not updatable - set from imported state
    freeform_tags   = { Environment = "prod" }
    defined_tags    = []
  }
  enable_auto_rotation {
    job_config_id = "<scheduler-resource-id>"
    key_source    = "ciphertrust"
  }
}
```

The example above shows all updatable attributes. Include only the attributes
that apply to your key - for example, omit `enable_auto_rotation` if the key
does not use scheduled rotation, or omit `oci_key_params.freeform_tags` and
`oci_key_params.defined_tags` if the key has no tags.

### 2. Import using the terraform import command

When `terraform import` is run it:

1. Reads the key from the API using the provided ID and stores the current values as state
2. Writes the imported state to the state file - no plan or apply is performed

Before running `terraform apply`, apply Option 1 or Option 2 above to avoid
unintended changes. Run `terraform plan` first to review exactly what would be
applied.

```shell
terraform import ciphertrust_oci_byok_key.example abc12345-1234-1234-1234-123456789012
```
