# Resource "ciphertrust_oci_byok_key_version" (terraform-provider-ciphertrust)

Use this resource to create and manage OCI BYOK key versions in CipherTrust Manager.

## Example Usage

```terraform
resource "ciphertrust_cm_key" "cm_key_version" {
  name       = "test-key-name"
  algorithm  = "AES"
  usage_mask = 60
}

# Add a BYOK key version to a native OCI key
resource "ciphertrust_oci_byok_key_version" "byok_version_0" {
  # Required parameters
  cckm_key_id                = ciphertrust_oci_key.test_key.id
  source_key_id              = ciphertrust_cm_key.cm_key_version.id
  # Optional parameters
  schedule_for_deletion_days = 14
  source_key_tier            = "local"
}

# Add a BYOK key version to a BYOK OCI key
resource "ciphertrust_oci_byok_key_version" "byok_version_0" {
  cckm_key_id     = ciphertrust_oci_byok_key.test_key.id
  source_key_id   = ciphertrust_cm_key.cm_key_version.id
  source_key_tier = "local"
}
```

## Import

An OCI BYOK key version can be imported using a composite ID made up of the
parent key's CipherTrust Manager resource ID and the version's CipherTrust
Manager resource ID separated by a dot:

`<cckm_key_id>.<version_id>`

### Important: Behaviour after import

After import, Terraform reads the version from the API and populates `id`,
`source_key_id`, `source_key_tier`, and all computed attributes in state.

The key version's attributes in OCI are fixed at creation and cannot be
changed. The `Update` operation sends no requests to OCI - only
`schedule_for_deletion_days` can differ between plan and state, and it is
applied locally at destroy time only. However, Terraform will still show plan
diffs for attributes that differ between the imported state and your resource
configuration. Use `ignore_changes` in Step 1 to keep the plan clean while
you inspect the imported values.

**`cckm_key_id`** - the CipherTrust Manager resource ID of the parent OCI
BYOK key. This is decoded from the composite import ID and set in state
automatically. Set this to the same value in your configuration.

**`schedule_for_deletion_days`** - this is a local setting only used when
the resource is destroyed. It defaults to `7` after import. Set this in your
configuration to the number of days you want before the version is deleted
when `terraform destroy` is run.

To avoid confusing plan output, do one of the following:

**Option 1 - Configure the resource block to match the key version's current**
**state** before running `terraform apply`. Set `cckm_key_id`, `source_key_id`,
`source_key_tier`, and `schedule_for_deletion_days` to their correct values so
that Terraform detects no diff.

**Option 2 - Use `lifecycle { ignore_changes }` to suppress diffs** on
attributes you want to inspect before configuring:

```terraform
lifecycle {
  ignore_changes = [
    source_key_id,
    source_key_tier,
  ]
}
```

The recommended approach (temporary use of `ignore_changes`):

1. Add the `lifecycle { ignore_changes }` block to the resource
2. Run `terraform apply` (or use `terraform import` for Terraform versions earlier than v1.5.0)
3. Run `terraform state show ciphertrust_oci_byok_key_version.example` to inspect the imported values
4. Update your resource configuration to match the imported state and set `schedule_for_deletion_days`
5. Remove `ignore_changes` so Terraform manages the attributes going forward

You can import a resource into Terraform in two ways.

### 1. Import using an import block and terraform apply

Only available for Terraform v1.5.0 and later.

When Terraform processes the `import` block it:

1. Reads the key version from the API using the provided ID and stores the current values as state
2. Compares that state to your resource block configuration
3. Applies any differences (in practice, Update sends no requests to OCI - only `schedule_for_deletion_days` is stored locally)

All three steps happen within a single `terraform apply` run.

**Step 1 - Import with `ignore_changes`** (safe starting point):

```terraform
import {
  to = ciphertrust_oci_byok_key_version.example
  id = "<cckm_key_id>.<version_id>"
}

resource "ciphertrust_oci_byok_key_version" "example" {
  cckm_key_id   = "<cckm_key_id>"
  source_key_id = "<cm-source-key-id>"
  lifecycle {
    ignore_changes = [
      source_key_id,
      source_key_tier,
    ]
  }
}
```

**Step 2 - Final config after inspecting state** (remove `ignore_changes`
once values are correct):

```terraform
resource "ciphertrust_oci_byok_key_version" "example" {
  cckm_key_id                = "<cckm_key_id>"
  source_key_id              = "<cm-source-key-id>"   # set from imported state
  source_key_tier            = "local"
  schedule_for_deletion_days = 14                     # set before running terraform destroy
}
```

### 2. Import using the terraform import command

When `terraform import` is run it:

1. Reads the key version from the API using the provided ID and stores the current values as state
2. Writes the imported state to the state file - no plan or apply is performed

Before running `terraform apply`, apply Option 1 or Option 2 above to avoid
confusing plan diffs. Run `terraform plan` first to review what would be
applied.

```shell
terraform import ciphertrust_oci_byok_key_version.example '<cckm_key_id>.<version_id>'
```
