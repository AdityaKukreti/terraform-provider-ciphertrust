# Resource "ciphertrust_aws_acl" (terraform-provider-ciphertrust)

Use this resource to create and manage AWS KMS access control lists (ACLs) in CipherTrust Manager.

## Example Usage

```terraform
# Define an ACL for a CipherTrust Manager user
resource "ciphertrust_aws_acl" "user_acl" {
  kms     = "ciphertrust-kms-id"
  user_id = "ciphertrust-user-id"
  actions = ["keycreate", "keyupdate", "keydelete"]
}

# Define an ACL for a CipherTrust Manager group
resource "ciphertrust_aws_acl" "group_acl" {
  kms     = "ciphertrust-kms-id"
  group   = "ciphertrust-group-name"
  actions = ["keycreate", "keyupdate", "keydelete"]
}
```

## Import

An AWS KMS ACL can be imported using a composite ID that encodes the KMS resource ID
and the user or group identity:

- User ACL:  `<kms-resource-id>::user::<user-id>`
- Group ACL: `<kms-resource-id>::group::<group-name>`

### Important: Behaviour after import

After import, Terraform reads the KMS's current ACL list and populates `actions`
with the permissions that are currently granted to the user or group.

If the `actions` set in your resource configuration differs from the imported
state, Terraform will grant or revoke the differing permissions on the next
`terraform apply`. Configure `actions` to match the current state before
running `terraform apply`, or use `lifecycle { ignore_changes = [actions] }` to
suppress any diff until you are ready to manage the ACL through Terraform.

The recommended approach (temporary use of `ignore_changes`):

1. Add `lifecycle { ignore_changes = [actions] }` to the resource
2. Run `terraform apply` (or use `terraform import` for Terraform versions earlier than v1.5.0)
3. Run `terraform state show ciphertrust_aws_acl.example` to inspect the imported actions
4. Update `actions` in your configuration to match the imported state
5. Remove `ignore_changes` so Terraform manages the ACL going forward

You can import a resource into Terraform in two ways.

### 1. Import using an import block and terraform apply

Only available for Terraform v1.5.0 and later.

When Terraform processes the `import` block it:

1. Reads the ACL from the API using the provided ID and stores the current values as state
2. Compares that state to your resource block configuration
3. Applies any differences as updates to the ACL (immediately during the same apply)

All three steps happen within a single `terraform apply` run.

**Step 1 - Import with `ignore_changes`** (safe starting point):

```terraform
import {
  to = ciphertrust_aws_acl.example
  id = "<kms-resource-id>::user::<user-id>"
}

resource "ciphertrust_aws_acl" "example" {
  kms     = "<kms-resource-id>"
  user_id = "<user-id>"
  actions = []
  lifecycle {
    ignore_changes = [actions]
  }
}
```

For a group ACL, replace `user_id = "<user-id>"` with `group = "<group-name>"` and
use the corresponding `::group::` composite ID.

**Step 2 - Final config after inspecting state** (remove `ignore_changes` once
actions are correct):

```terraform
resource "ciphertrust_aws_acl" "example" {
  kms     = "<kms-resource-id>"
  user_id = "<user-id>"
  actions = ["keycreate"]
}
```

### 2. Import using the terraform import command

When `terraform import` is run it:

1. Reads the ACL from the API using the provided ID and stores the current values as state
2. Writes the imported state to the state file - no plan or apply is performed

Before running `terraform apply`, add `lifecycle { ignore_changes = [actions] }` to
avoid unintended grant or revoke operations. Run `terraform plan` to review what
would be applied.

```shell
# User ACL
terraform import ciphertrust_aws_acl.example '<kms-resource-id>::user::<user-id>'

# Group ACL
terraform import ciphertrust_aws_acl.example '<kms-resource-id>::group::<group-name>'
```
