## What this repo is

The **CipherTrust Terraform Provider** lets users manage Thales CipherTrust Manager (CM)
resources using Infrastructure-as-Code. It translates `terraform plan/apply/destroy` calls
into CipherTrust Manager REST API calls. It is published at
`github.com/ThalesGroup/terraform-provider-ciphertrust` on branch `1.0.1`.

CipherTrust Manager (CM) is Thales's enterprise key-management platform. Provider
sub-areas: **CM** (core platform), **CCKM** (cloud key-management — AWS/Azure/GCP/OCI),
**CTE** (CipherTrust Transparent Encryption — file/volume encryption policies).

## ⚠ Critical project constraint — TFIN-174

**Empty `Read()` is the root cause of ALL drift issues.** Every resource's `Read` method
currently returns without calling the CM API, so Terraform never reconciles state against
reality. This masks all other bugs (immutable-field violations, case mismatches, tag swaps).

**Fix order is mandatory:**
1. Fix `Read()` first (TFIN-174) — once API is called during plan, all drift becomes visible.
2. Fix `RequiresReplace` on immutable fields (TFIN-183, TFIN-179, TFIN-181).
3. Fix case normalization and JSON tag issues (TFIN-195, TFIN-194).

If your task is NOT TFIN-174, do not implement a proper `Read()` as a side-effect — it will
surface drift before the immutable-field fixes are in and will confuse users. Leave `Read()`
empty unless the ticket specifically asks you to fix it.

# Implementation Plan: TFIN-286 — Fix swapped JSON tags for `revocation_reason` / `revocation_message`

## Root cause

In `internal/provider/cm/schema_cm.go`, the `CMKeyJSON` struct declares the JSON struct tags for `RevocationReason` and `RevocationMessage` in swapped form. The Go field `RevocationReason` currently carries `json:"revocationMessage,omitempty"` and the Go field `RevocationMessage` currently carries `json:"revocationReason,omitempty"`. As a result, on Create/Update the Terraform attribute `revocation_reason` is transmitted to CipherTrust Manager under the wire key `revocationMessage`, and vice versa. CM persists both values under the wrong keys.

This is a pure serialization bug — no schema, no plan-to-model assignment, and no CRUD control flow needs to change beyond the tag correction itself (see §"Files to MODIFY" item 2 for the definitive determination).

## Files to MODIFY

### 1. `internal/provider/cm/schema_cm.go` (the fix)

- Locate the `CMKeyJSON` struct.
- Change the tag on Go field `RevocationReason` from `json:"revocationMessage,omitempty"` to `json:"revocationReason,omitempty"`.
- Change the tag on Go field `RevocationMessage` from `json:"revocationReason,omitempty"` to `json:"revocationMessage,omitempty"`.
- Preserve `omitempty` exactly; do not introduce other tag options. Do not rename Go fields.
- Reference exemplar for the corrected `snake_case` ↔ `camelCase` tag convention: the `Name` field (`json:"name,omitempty"`) and `ObjectType` field (`json:"objectType,omitempty"`) on the same `CMKeyJSON` struct — both are unambiguous one-to-one tag pairs whose Terraform schema counterparts (`name`, `object_type`) follow the identical naming rule this fix restores.

### 2. `internal/provider/cm/resource_cm_key.go` (definitive verification, no expected change)

- Inspect `Create` and `Update`. The plan-to-JSON mapping uses the Go field names (`payload.RevocationReason = plan.RevocationReason.ValueString()` and the analogous line for `RevocationMessage`). Because the bug lives entirely in the JSON tags and the Go field names were never swapped, the Create/Update assignment statements are already correct and **must not** be modified. Modifying them would re-introduce the swap at the model layer.
- Do **not** modify `Read()` — TFIN-174 mandates Read remains empty.
- Do not touch `Delete` or schema definitions.

### 3. JSON response decode-path audit (verification, no expected change)

- `CMKeyJSON` is also used to decode CM responses in `Create` (to populate computed attributes such as `id`, `uri`, `state`). Because the response decode uses the same struct, prior-version resources whose CM-side data is currently stored under swapped keys will, after this fix, decode those stale values *consistently swapped* on the next refresh — but since Read is a no-op (TFIN-174), no state is populated from the response decode for revocation fields today. Therefore no compensating remediation code is in scope. Document this in the changelog entry (see file 5).

### 4. `internal/provider/cm/resource_cm_key_test.go` (new or extended acceptance test)

- Add acceptance test function `TestAccCMKey_revocationFieldsRoundTrip`.
- Add unit test function `TestCMKeyJSON_RevocationTagsMarshal` that constructs a `CMKeyJSON` literal with distinct sentinel strings for `RevocationReason` ("REASON_SENTINEL") and `RevocationMessage` ("MESSAGE_SENTINEL"), calls `json.Marshal`, and asserts the resulting bytes contain `"revocationReason":"REASON_SENTINEL"` and `"revocationMessage":"MESSAGE_SENTINEL"`. A symmetric `TestCMKeyJSON_RevocationTagsUnmarshal` asserts the inverse.
- Pattern reference: existing unit-style marshal tests under `internal/provider/cm/` (mirror the file/function naming convention of any existing `*_test.go` in that package); if no precedent exists in `internal/provider/cm/`, follow the naming pattern used in `internal/provider/cte/` test files.

### 5. `CHANGELOG.md` (in scope)

- Add an entry under the next unreleased version:
  - `BUG FIXES: ciphertrust_cm_key — revocation_reason and revocation_message JSON tags were swapped, causing values to be persisted under the wrong keys on CipherTrust Manager (TFIN-286). After upgrading, the next apply that modifies either attribute will write both values correctly. Resources not re-applied will retain swapped CM-side storage; manual remediation is out of scope.`
