## What this repo is

The **CipherTrust Terraform Provider** lets users manage Thales CipherTrust Manager (CM)
resources using Infrastructure-as-Code. It translates `terraform plan/apply/destroy` calls
into CipherTrust Manager REST API calls. It is published at
`github.com/ThalesGroup/terraform-provider-ciphertrust` on branch `1.0.1`.

CipherTrust Manager (CM) is Thales's enterprise key-management platform. Provider
sub-areas: **CM** (core platform), **CCKM** (cloud key-management — AWS/Azure/GCP/OCI),
**CTE** (CipherTrust Transparent Encryption — file/volume encryption policies).

## ⚠ Critical project constraint — TFIN-174

**Empty `Read()` is the root cause of ALL drift issues.** Every resource's `Read` method
currently returns without calling the CM API, so Terraform never reconciles state against
reality. This masks all other bugs (immutable-field violations, case mismatches, tag swaps).

**Fix order is mandatory:**
1. Fix `Read()` first (TFIN-174) — once API is called during plan, all drift becomes visible.
2. Fix `RequiresReplace` on immutable fields (TFIN-183, TFIN-179, TFIN-181).
3. Fix case normalization and JSON tag issues (TFIN-195, TFIN-194).

If your task is NOT TFIN-174, do not implement a proper `Read()` as a side-effect — it will
surface drift before the immutable-field fixes are in and will confuse users. Leave `Read()`
empty unless the ticket specifically asks you to fix it.

# Implementation Plan: TFIN-286 — Fix swapped JSON tags for revocation_reason / revocation_message

## Root cause

In `internal/provider/cm/schema_cm.go`, the `CMKeyJSON` struct has its `json:"..."` struct tags swapped on the two revocation fields. The Go field `RevocationReason` is currently tagged `json:"revocationMessage,omitempty"` and `RevocationMessage` is tagged `json:"revocationReason,omitempty"`. Create/Update payloads therefore send the reason as the message and vice versa. The Terraform schema attribute names (`revocation_reason`, `revocation_message`) are user-facing and remain unchanged.

## Files to MODIFY

### 1. `internal/provider/cm/schema_cm.go` (primary fix)

In the `CMKeyJSON` struct, apply this exact tag swap (Go field identifiers `RevocationReason` and `RevocationMessage` are correctly named per the ticket — **do NOT rename them**, swap tags only):

- `RevocationReason string` → tag becomes `json:"revocationReason,omitempty"`
- `RevocationMessage string` → tag becomes `json:"revocationMessage,omitempty"`

Per the CipherTrust Manager API contract (confirmed via the existing key-revocation handling already present in the provider and CM's published key schema), `revocationReason` is the enumerated reason code (`Unspecified`, `KeyCompromise`, `CACompromise`, `AffiliationChanged`, `Superseded`, `CessationOfOperation`, `PrivilegeWithdrawn`) and `revocationMessage` is the free-form description. No further spec lookup needed; the existing schema validators in `resource_cm_key.go` already encode the enum on `revocation_reason`, which itself confirms the intended mapping.

Do not change `omitempty`, field order, or any surrounding fields.

### 2. `internal/provider/cm/resource_cm_key.go` (verify only, no edits expected)

Inspect the resource type's `Create` and `Update` methods (the `Resource` interface implementations on the `resourceCMKey` struct) and any helper that builds the request body (e.g. `buildCMKeyJSONFromPlan` / inline assignments inside `Create`/`Update`). Confirm the assignments read:

- `body.RevocationReason = plan.RevocationReason.ValueString()`
- `body.RevocationMessage = plan.RevocationMessage.ValueString()`

Since the Go field names were correct all along, no code change is required here — only the tags were wrong. **Do NOT modify `Read()`** (TFIN-174: all `Read()` methods are intentionally no-op).

## Pattern to follow

Reference struct: **`CMUserJSON`** in `internal/provider/cm/schema_cm.go`, used by `internal/provider/cm/resource_cm_user.go`. Each snake_case Terraform attribute maps one-to-one to a Go struct field whose `json:"..."` tag is the exact CM API camelCase name. Use this struct as the alignment template when verifying the fix.

## Schema changes

**None.** Both `revocation_reason` and `revocation_message` remain `types.StringType`, Optional. No `RequiresReplace`, no plan modifiers, no migration.
