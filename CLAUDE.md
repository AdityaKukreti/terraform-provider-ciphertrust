## What this repo is

The **CipherTrust Terraform Provider** lets users manage Thales CipherTrust Manager (CM)
resources using Infrastructure-as-Code. It translates `terraform plan/apply/destroy` calls
into CipherTrust Manager REST API calls. It is published at
`github.com/ThalesGroup/terraform-provider-ciphertrust` on branch `1.0.1`.

CipherTrust Manager (CM) is Thales's enterprise key-management platform. Provider
sub-areas: **CM** (core platform), **CCKM** (cloud key-management ‚Äî AWS/Azure/GCP/OCI),
**CTE** (CipherTrust Transparent Encryption ‚Äî file/volume encryption policies).

## ‚öÝ Critical project constraint ‚Äî TFIN-174

**Empty `Read()` is the root cause of ALL drift issues.** Every resource's `Read` method
currently returns without calling the CM API, so Terraform never reconciles state against
reality. This masks all other bugs (immutable-field violations, case mismatches, tag swaps).

**Fix order is mandatory:**
1. Fix `Read()` first (TFIN-174) ‚Äî once API is called during plan, all drift becomes visible.
2. Fix `RequiresReplace` on immutable fields (TFIN-183, TFIN-179, TFIN-181).
3. Fix case normalization and JSON tag issues (TFIN-195, TFIN-194).

If your task is NOT TFIN-174, do not implement a proper `Read()` as a side-effect ‚Äî it will
surface drift before the immutable-field fixes are in and will confuse users. Leave `Read()`
empty unless the ticket specifically asks you to fix it.

# Implementation Plan: TFIN-286 (Revised)

## Root cause

In `internal/provider/cm/schema_cm.go`, the `CMKeyJSON` struct has swapped JSON tags on its two revocation fields. Field `RevocationReason` currently serializes as `revocation_message`, and `RevocationMessage` currently serializes as `revocation_reason`. Create/Update payloads therefore write each value to the wrong CipherTrust Manager API field, producing latent CM-side corruption and (once Read is implemented per TFIN-174) perpetual drift.

## Files to MODIFY

### 1. `internal/provider/cm/schema_cm.go` (primary fix)

Locate the `CMKeyJSON` struct. The two affected fields today carry these literal struct tags:

- `RevocationReason string` tagged `` `json:"revocation_message,omitempty"` ``
- `RevocationMessage string` tagged `` `json:"revocation_reason,omitempty"` ``

Swap the tag key names so they become:

- `RevocationReason string` tagged `` `json:"revocation_reason,omitempty"` ``
- `RevocationMessage string` tagged `` `json:"revocation_message,omitempty"` ``

`omitempty` is confirmed present on both fields today and must be retained. Do not rename the Go identifiers ‚Äî `resource_cm_key.go` references them by Go name and must compile unchanged.

**Pattern reference (in-file):** sibling fields in the same `CMKeyJSON` struct already use the snake_case CM REST convention with `omitempty`, e.g. `Name string` ‚Üí `` `json:"name,omitempty"` ``, `Algorithm string` ‚Üí `` `json:"algorithm,omitempty"` ``, `UsageMask int` ‚Üí `` `json:"usageMask,omitempty"` ``. The corrected revocation tags follow the same form.

### 2. Terraform schema ‚Äî no changes

The two affected Terraform attributes in `internal/provider/cm/resource_cm_key.go` are already correct and stay as-is:

- `revocation_reason` ‚Äî type `types.StringType`, **Optional**, not Computed.
- `revocation_message` ‚Äî type `types.StringType`, **Optional**, not Computed.

No `RequiresReplace` modifier is added; no Required/Optional/Computed flag changes.

### 3. Read() ‚Äî no changes

Per TFIN-174, the `Read()` method in `internal/provider/cm/resource_cm_key.go` remains an intentional no-op. This ticket does not modify it.

### 4. Downstream compile-check surface

Confirm that the only callers of `CMKeyJSON.RevocationReason` and `CMKeyJSON.RevocationMessage` are the Create and Update marshal paths in `internal/provider/cm/resource_cm_key.go` (the `populate*` / `build*Payload` helpers and their inverse in any state-mapping helpers). Because the Go field names are unchanged, no other file requires edits. Verify with `grep -rn "RevocationReason\|RevocationMessage" internal/provider/cm/`.

## Files to CREATE / EXTEND for tests

### `internal/provider/cm/schema_cm_test.go` (create or extend)

Add a table-driven unit test named `TestCMKeyJSON_RevocationFieldSerialization` that:

- Constructs `CMKeyJSON` values with `RevocationReason="Unspecified"` and `RevocationMessage="audit note"`.
- Calls `encoding/json.Marshal`.
- Asserts the output contains `"revocation_reason":"Unspecified"` and `"revocation_message":"audit note"`, and does NOT contain the swapped form.
- Includes a zero-value case asserting both keys are omitted (verifies `omitempty` is preserved).

**Pattern reference:** follow the existing table-driven marshal test style in `internal/provider/cm/schema_cm_test.go` if present; otherwise mirror the table-driven structure used in `internal/provider/cm/resource_cm_user_test.go` (uses `[]struct{ name string; in ‚Ä¶; want string }` with `t.Run`).
